## Scenario

Organizations frequently enforce policies that restrict account activity to designated geographic regions — guarding against account sharing, unauthorized access from foreign locations, and use of non-corporate VPNs. Impossible travel detection is a technique that flags authentication anomalies where a user account logs in from two or more geographically distant locations within a timeframe that would be physically impossible to travel between.

A Microsoft Sentinel analytics rule was configured to detect this pattern by monitoring sign-in logs for accounts whose logon sequences exceed a defined geographic dispersion threshold within a set time window. The rule triggered an incident requiring investigation.

### 1. Preparation

Prior to investigation, roles and scope were established:

Detection logic was built around the business context that logins occurring from geographically distant regions within an implausible timeframe represent either:
- A compromised credential being used by a threat actor in a different location
- Account sharing in violation of corporate policy
- Use of a VPN or proxy masking true origin

---

### 2. Detection & Analysis

#### KQL Detection Logic

```kql
let TimeDelta = 4h;
let DistanceThreshold = 500;
SigninLogs
| where ResultType == 0
| extend City = tostring(LocationDetails.city)
| extend CountryOrRegion = tostring(LocationDetails.countryOrRegion)
| extend Latitude = tostring(LocationDetails.geoCoordinates.latitude)
| extend Longitude = tostring(LocationDetails.geoCoordinates.longitude)
| project TimeGenerated, UserPrincipalName, IPAddress, City, CountryOrRegion, Latitude, Longitude
| sort by UserPrincipalName, TimeGenerated asc
| serialize
| extend PrevTime = prev(TimeGenerated), PrevCity = prev(City), PrevCountry = prev(CountryOrRegion), PrevUser = prev(UserPrincipalName)
| where UserPrincipalName == PrevUser
| extend TimeDiff = datetime_diff('minute', TimeGenerated, PrevTime)
| where TimeDiff between (0 .. (toint(TimeDelta / 1m)))
| where City != PrevCity
| project UserPrincipalName, IPAddress, City, PrevCity, CountryOrRegion, TimeDiff
| order by TimeDiff asc
```
The Sentinel analytics rule triggered an active incident. Upon investigation, **55 accounts** were flagged as exhibiting logon patterns that exceeded the defined geographic dispersion threshold.

---

### 3. Containment & Eradication

The following response procedures apply upon confirmation of a true-positive impossible travel event:

- **Isolate** the affected endpoint via Microsoft Defender for Endpoint if device-level compromise is suspected
- **Disable the account** in Microsoft Entra ID immediately to prevent further unauthorized access
- **Contact the user or their manager** to determine whether the activity was legitimate (e.g., authorized travel, VPN use)
- **Review all recent activity** associated with the account — file access, email forwarding rules, privilege escalation attempts
- **Revoke active sessions** and force token refresh across all connected applications
- **Remove any persistence mechanisms** established during the unauthorized session
- **Restore affected systems** to normal operations following confirmation of eradication

---

### 4. Post-Incident Activity

- **Policy update:** Review and strengthen geographic access policies to explicitly define approved regions and acceptable use of VPNs
- **Geo-fencing implementation:** Propose Conditional Access policy in Azure Entra ID to restrict or require MFA for logins originating outside approved geographic regions
- **Tuning:** Refine the Sentinel analytics rule thresholds (time window, distance threshold) to reduce false positive volume while maintaining detection sensitivity

## Key Takeaways

- Impossible travel detection is a high-value, low-complexity rule that surfaces credential compromise and policy violations with minimal tuning overhead
- 55 flagged accounts in a single incident highlights the importance of risk-based triage — not every alert warrants the same depth of investigation
- Geo-fencing via Conditional Access is a strong preventive control that complements detective rules like this one
- Entra ID sign-in logs combined with KQL provide granular visibility into authentication patterns that EDR alone cannot surface
